# Plan: Payment Recovery + Referral Program

Implements both request batches (Part A: dunning/cancellation, Part B: referrals) in one migration+code pass. Existing `checkout.session.completed` and `invoice.paid` handlers left untouched.

## Part A — Failed Payments & Cancellation

### A1. Schema on `companies`
Migration adds:
- `payment_status text default 'current'` — CHECK in (`current`,`past_due`,`canceled`,`suspended`) (suspended included now so A3 doesn't need a re-migration)
- `payment_failed_at timestamptz`
- `grace_period_ends_at timestamptz`
- `dunning_reminders_sent int[] default '{}'` — day-marks already fired (0/3/6), idempotent
- `cancellation_reason text`, `cancellation_feedback text`
- Index on `payment_status`

`check-subscription/index.ts` already writes tier/status fields on `companies`; we extend the same row (no duplicate table).

### A2. Webhook expansion (`stripe-webhook/index.ts`)
Add cases, following the existing `emitToOrchestrator` + `company_id` lookup pattern. Introduce a shared service-role Supabase client (currently `_supabase` is unused) to write status updates.

- `invoice.payment_failed` → set `payment_status='past_due'`, `payment_failed_at=now()`, `grace_period_ends_at=now()+7d`, reset `dunning_reminders_sent='{}'`; emit `payment_failed`.
- `invoice.payment_succeeded` → reset `payment_status='current'`, clear failed/grace/dunning fields; emit `payment_recovered`. Keep existing `invoice.paid` case intact (both fire; the reset is idempotent).
- `customer.subscription.deleted` → `payment_status='canceled'`; emit `subscription_canceled`.

### A3. Dunning cron (`dunning-reminder-scan`)
New edge function modeled on `winback-scan` (cron secret auth) + `send-company-welcome` (Resend, `FROM = 'Aura Intercept <ai@auraintercept.ai>'`).

Logic:
- Query companies where `payment_status='past_due'`.
- For each, compute `daysSince = floor((now - payment_failed_at)/day)`.
- For each threshold in `[0,3,6]` where `daysSince >= threshold` and threshold not yet in `dunning_reminders_sent`: send the corresponding email (day 0 = card declined, day 3 = X days left, day 6 = final notice tomorrow), append threshold to array.
- Each email CTAs to a portal link generated by invoking `stripe-customer-portal` with a service-role auth context.
- If `now > grace_period_ends_at` and still `past_due`: set `payment_status='suspended'`, emit `subscription_suspended`. No feature-gating in this pass (per prompt).

Register `aura-dunning-reminders` cron (daily) using the same pg_cron pattern as other `aura-*` jobs.

### A4. Cancellation feedback
- New component `CancelSubscriptionDialog` shown from `Subscription.tsx` before invoking `stripe-customer-portal`. Required dropdown (5 reasons) + optional textarea. On submit: write to `companies.cancellation_reason` / `cancellation_feedback`, then continue to portal redirect.
- `Analytics.tsx` "Signup Funnel" tab (platform_admin only) gets a new adjacent section "Cancellation Reasons" showing grouped counts from `companies` where `cancellation_reason is not null`.

## Part B — Referral Program

### B1. `referrals` table + code function
Migration:
```
referrals(
  id uuid pk default gen_random_uuid(),
  referring_company_id uuid not null references companies(id),
  referral_code text not null unique,
  referred_email text,
  referred_company_id uuid references companies(id),
  status text not null default 'pending'
    check (status in ('pending','signed_up','converted','rewarded')),
  created_at timestamptz default now(),
  converted_at timestamptz,
  rewarded_at timestamptz
)
```
GRANTs to authenticated + service_role. RLS: company admins can SELECT/INSERT where `referring_company_id = get_user_company_id(auth.uid())`; platform_admin sees all via `has_role`.

`get_or_create_referral_code(p_company_id uuid) returns text` SECURITY DEFINER — code = `encode(gen_random_bytes(4),'hex')` (8 chars), retry on unique collision, returns existing if any.

### B2. Referral card in dashboard
New `ReferralCard` component embedded in `Subscription.tsx` (billing page). Calls `get_or_create_referral_code`, displays `https://auraintercept.ai/for-business?ref=<code>`, Copy Link + Share via SMS (`sms:?body=...`) buttons, and a status list of that company's referrals.

### B3. Wire ref through signup + reward on first checkout
- `ForBusiness.tsx`: read `?ref=` from URL, propagate into `startLiveDemo()` /auth URL alongside industry/tier.
- `SignUp.tsx`: on `signup_completed` success branch, if `ref` present, look up `referrals` by code; set `referred_company_id`, `status='signed_up'`.
- `create-checkout/index.ts`: inside the existing `isFirstCheckout` block, look up any `referrals` row with `referred_company_id = companyId`. If found:
  - Update to `status='converted'`, `converted_at=now()`.
  - Waive the new company's onboarding fee (set `onboardingWaivedReason='referral'`, reuse the same waived-status write path).
  - Fetch referring company's `stripe_customer_id` + active subscription; create a 100%-off one-month Stripe coupon and apply via `stripe.subscriptions.update(subId, { coupon })`.
  - On success: `status='rewarded'`, `rewarded_at=now()`.
  - On any failure: `logStep` clearly, leave at `converted` for manual follow-up. Never block checkout.
